## External review finding

- Migrated 0.21.0 schedule rows intentionally keep
  `laboratory_location_id = NULL`; no identity is inferred from their labels.
- The current location-conflict lookup compares exact stable IDs. An active
  overlapping historical row can therefore fail to reserve its unknown
  physical place when a different operator is assigned to the new item.
- Accepting that reservation would silently assume two physical locations are
  distinct without evidence.
- Selected conservative behavior: every overlapping non-terminal schedule row
  without stable location identity blocks creation or movement until an
  operator explicitly identifies its real location from the existing stable
  location choices.
- Conflict priority will remain deterministic: same operator, unresolved
  historical location, then same stable location ID.
