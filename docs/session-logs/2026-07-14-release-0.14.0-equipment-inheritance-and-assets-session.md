# 2026-07-14 - Release 0.14.0 Equipment Inheritance And Assets Session

## Objective

Correct the Equipment Repository UX around shared fields, required semantics,
documents, serial-numbered physical materials, and redundant preview tabs.

## Work Completed

- Added equipment migration 5 with the editable universal `Général` category.
- Moved common field rules to `Général` while preserving family
  `root_category_id` classification and catalog filters.
- Archived the provisional model-level variant and documentation-URL defaults.
- Added typed file-reference validation in Rust core and content-addressed
  local file storage in the Rust agent.
- Added the loopback equipment-file route and tests for safe filenames, digest
  evidence, object persistence, and model revision references.
- Added LAB CONSOLE field editing/archival, clear required/optional badges,
  category-form requirements, uploaded-file controls, and one preview with
  repliable technical evidence.
- Added LAB CONSOLE physical-material registration through the metrology API
  from an approved equipment model, with serial number, part number,
  serviceability, and calibration policy.
- Added metrology migration 8 and an explicit equipment-model identity,
  revision, and checksum reference so both category systems remain distinct.
- Added frontend and backend regressions for the complete paths.

## Modeling Notes

- `Général` participates in parent traversal but is not returned as a selectable
  equipment family in the model wizard.
- Manufacturer/model remain structural and non-archivable. Serial number is
  deliberately confined to the physical asset record.
- Uploaded bytes are outside SQLite; the revisioned equipment definition,
  audit payload, and outbox contain the typed SHA-256 manifest.
- A catalog-derived physical asset uses the Equipment Repository hierarchy as
  its classification evidence; it does not inject equipment category codes
  into the separate metrology category foreign key.
