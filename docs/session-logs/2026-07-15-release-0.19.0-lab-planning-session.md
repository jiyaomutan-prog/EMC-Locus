# 2026-07-15 - Release 0.19.0 Lab Planning Session

## Objectif

Livrer la première verticale utilisable de Locus Lab Management : ouvrir ou
créer un dossier d'essai, terminer sa revue du besoin, le faire passer en
planification et réserver un premier créneau audité sans contourner le Local
Agent Rust.

## Baseline

- version : `0.18.0` ;
- branche de départ : `main` ;
- commit de départ : `5bc32c35f8cafb5e680acee83284d1b88b436e01` ;
- branche de travail : `feature/0.19.0-lab-planning` ;
- état initial : propre et synchronisé avec `origin/main`.

## Audit initial

- Les routes Rust permettent déjà de créer et lire un projet, compléter sa
  revue du besoin et passer en planification.
- La table `service_schedule_items` et de nombreuses règles existent en Python,
  mais le document API les qualifie encore de stockage legacy.
- Cette écriture directe contredit la cible où le Local Agent est l'unique
  propriétaire des transactions, de l'audit et de l'outbox.
- LAB CONSOLE n'expose aucun dossier projet et affiche encore Planification,
  Campagnes et Rapports comme « à venir ».
- La Test Station possède déjà son parcours de préparation ; ajouter une autre
  bibliothèque technique avant de fermer le parcours projet serait une mauvaise
  priorité produit.

## Phases

### Phase 1 - Workflow opérateur

État : terminée.

- Parcours formalisé dans
  `docs/ux/0.19.0-lab-planning-workflow.md`.
- Rôle, objet métier, étapes, décisions, états, erreurs, résultat attendu et
  critères de refus UX documentés avant l'implémentation.

### Phase 2 - Domaine et agent

État : terminée.

- `ServiceScheduleItem` et `ServiceScheduleStatus` typés dans le core avec
  dates locales canoniques, journée ouvrée, ordre des transitions, recouvrement
  et conflits opérateur/lieu.
- Migration projets `6` ajoutant révision optimiste, auteur de création et
  dernier auteur, sans modifier les migrations historiques.
- Création, lecture et transitions de planning portées derrière l'agent Rust.
- Routes contextuelles au dossier avec actions `confirm`, `start`, `complete`
  et `cancel`.
- Idempotence, concurrence optimiste, audit projet et outbox atomiques.
- Client Python aligné sur le nouveau contrat.

### Phase 3 - LAB CONSOLE

État : terminée.

- Client TypeScript ajouté pour les dossiers, la revue du besoin, le planning
  et l'historique projet.
- Vue opérateur ajoutée autour d'un dossier sélectionné, de sa prochaine action,
  de sa checklist et de ses créneaux.
- Création de dossier et réservation de créneau dans des tiroirs contextualisés,
  sans exposer les identifiants d'opération.
- Conflits d'opérateur et de lieu traduits en refus métier dans le formulaire
  qui les a produits.
- Revue terminée compactée après passage en planification afin de garder le
  créneau visible à 1280 x 720.
- Navigation future supprimée ; seuls les espaces réellement utilisables sont
  présentés comme actifs.
