# 2026-07-14 - Release 0.18.0 Correction Workflow Session

## Contexte

Le cahier des charges joint visait `0.16.0` depuis `0.15.0`, mais `main` etait
deja propre et valide en `0.17.0` au commit `d622131`. La passe a donc ete
livree comme la release coherente suivante, `0.18.0`, sans retrograder la
baseline ni reecrire l'historique publie.

## Travail realise

- Revue UX prealable des ecrans et de cinq cas physiques : cable RF, antenne,
  amplificateur, sonde de courant et accelerometre IEPE.
- Separation typee des exigences de correction, valeurs nominales, sources
  propres au numero de serie et affectations revues.
- Service/repository SQLite avec concurrence optimiste, idempotence,
  approbation, activation, remplacement, audit et outbox.
- Resolveur contextuel avec validite, politique de repli, contexte accredite ou
  simulation et conditions de polarisation.
- Routes HTTP et client Python pour creation, lecture, revue, transitions,
  historique et resolution.
- Refonte du dossier materiel LAB CONSOLE et de l'editeur de chemins du modele.
- Import CSV, graphe, preuve locale, validite, environnement, avant/apres
  reglage, revue et comparaison nominale/etalonnée.
- Correction de l'unite operateur `m_per_s2` en `m/s²`.
- Selection automatique du materiel nouvellement enregistre.
- Suppression de la presentation factuellement fausse de la metrologie comme
  module a venir.
- Remplacement du lanceur E2E destructif par un agent et un stockage isoles.
- Isolation equivalente du smoke des lanceurs par stockage temporaire et etat
  de processus nomme, sans toucher a `data/local-agent` ni a l'agent actif.

## Autocontroles et corrections

- Le premier lancement E2E n'initialisait pas les migrations : le harnais
  initialise maintenant explicitement son depot temporaire.
- Les assertions certificat etaient ambiguës car la meme preuve est visible
  dans le verdict et le detail : les selecteurs sont scopes aux regions
  accessibles.
- L'ajout des nouveaux E2E a expose deux hypotheses d'ordre dans le scenario de
  classification : le test ouvre maintenant le formulaire et le produit
  selectionne le nouvel enregistrement.
- Le test de migration Python attendait encore metrology `1..9` : il couvre
  maintenant `1..10`.
- La revue transactionnelle a montre que le remplacement rendait l'ancienne
  affectation obsolete sans evenement propre : la supersession emet maintenant
  son audit et son message outbox dans la meme transaction que l'activation.
- La verification avant CI a detecte que le smoke historique employait encore
  `data/local-agent -Reset` : les lanceurs acceptent maintenant un stockage et
  un nom d'etat isoles, utilises par le smoke avec nettoyage borne.
- Le premier smoke isole a encore rencontre le binaire `target/debug` verrouille
  par l'agent actif : son build Cargo utilise maintenant un `target` temporaire
  propre, lui aussi nettoye apres verification de son chemin.
- Les captures ont ete controlees comme PNG opaques `24bpp RGB`, avec les
  dimensions exigees et sans unite interne brute.

Le script CI local a valide Rust, Python et les migrations, puis le wrapper
`pnpm` fourni par l'environnement Codex a refuse `npm run` en tentant une
installation avec scripts ignores. Les memes outils frontend ont donc ete
executes directement avec Node, sans installation : TypeScript, ESLint,
Vitest, Vite et Playwright sont tous passes. Le statut GitHub Actions reste a
ajouter au compte rendu de fin apres commit et push.

## Limites maintenues

- Pas d'application numerique de la correction au runtime.
- Pas de nouveau moteur d'acquisition, FFT, execution, reporting ou RBAC.
- Les identites authentifiees et competences d'approbation restent a modeliser.
- Les anciennes routes techniques de conversions/courbes ne sont pas renommees
  dans cette passe ; elles ne sont pas exposees comme vocabulaire operateur.
