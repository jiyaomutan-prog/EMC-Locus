# Release 0.15.0 - Equipment Signals And Controlled Corrections

Version `0.15.0` replaces ambiguous measurement-engineering terminology with a
model based on the physics and representation of the signal.

## Delivered

- Equipment models declare readable input/output ports and signal paths from a
  measurement input to a physical or logical result output.
- Signal paths reference approved sample conversions and frequency responses by
  stable identity, revision id, and canonical definition checksum.
- Time-domain sample conversions explicitly cover gain, offset, lookup and
  controlled expression methods, with optional overload/clipping bounds.
- Frequency responses require one amplitude component, support one optional
  phase component, and declare how each value is applied rather than relying on
  naming or sign conventions.
- LAB CONSOLE replaces `Ingénierie de mesure`, `Scaling`, `Courbes
  d’ingénierie`, and `Excitation` with `Signaux et corrections`, `Conversions
  temporelles`, `Réponses fréquentielles`, and `Alimentation / conditionnement
  du capteur`.
- Input/output cards expose ordinary model editing, while low-level electrical
  details remain available in a focused advanced disclosure.
- The Rust agent blocks unknown, unapproved, or checksum-mismatched correction
  references on every model creation, edit, submission, and approval path.

## Model Decisions

- A time-domain conversion operates on sampled values. A frequency response
  operates on spectral amplitude and optional phase. They are separate
  contracts even when both originate from the same certificate.
- Sensor power/conditioning describes IEPE current, bridge supply, charge
  conditioning, or equivalent measurement-chain needs. It is not the stimulus
  applied to the equipment under test.
- Frequency correction operation is explicit. For example, an RF cable loss
  table can add positive dB loss to a measured level over its Fmin/Fmax band.
- A correction reference is controlled evidence. An editable label or an
  unversioned JSON fragment is not sufficient for an approved model.
- Existing physical assets already pin their equipment-model revision and
  checksum. Per-asset correction override and station wiring remain separate
  future domains so metrology decisions are not hidden in mutable asset text.

## Limits

- The release defines, validates, persists, and displays correction contracts;
  it does not apply them to live values.
- It does not add real acquisition, FFT, phase processing, station wiring, or a
  graphical chain editor.
- It does not yet let a metrologist derive an asset-specific correction from a
  newly uploaded calibration certificate.
- Authentication, RBAC, reporting, and central synchronization remain outside
  this release.

## Validation

The release is validated with Rust format, Clippy and workspace tests; Python
compilation, unit tests and SQLite migration validation; LAB CONSOLE typecheck,
lint, unit tests, production build and Playwright E2E; release consistency,
browser inspection, and Git whitespace checks.
