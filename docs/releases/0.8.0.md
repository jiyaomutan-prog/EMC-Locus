# EMC Locus 0.8.0 - Simulated EMC Execution Preflight

## Objective

Version `0.8.0` delivers the next vertical slice after the metrology baseline:
one complete simulated EMC execution workflow with blocking metrology preflight.

The workflow is intentionally narrow. It does not add real instrument drivers,
acquisition, FFT, reporting, planning reservations, or file upload. It proves
that a test-station operator can launch an attempt, receive an explainable
refusal when instrumentation is not ready, correct the metrology state, launch
again, persist the simulated result, and inspect audit/outbox evidence.

## Delivered

1. New project migration `0003_simulated_test_executions.sql`.
2. New persisted aggregate `simulated_test_executions`.
3. New per-instrument snapshot table
   `simulated_test_execution_instruments`.
4. Dedicated Rust agent service, repository, and DTO boundary for simulated EMC
   execution.
5. Dedicated API routes:

```text
POST /api/v1/test-executions/simulated-emc
GET  /api/v1/test-executions/{attempt_id}
GET  /api/v1/projects/{code}/test-executions
```

6. Test-context metrology readiness call using project, attempt, method, mode,
   checked date, and required instruments.
7. Structured refusal persisted as valid evidence:

```text
code
message
causes[]
causes[].asset_id
causes[].dimension
causes[].code
causes[].message
```

8. Refusal dimensions now distinguish:

```text
serviceability
missing_evidence
calibration_validity
nonconformance
```

9. Deterministic simulated conducted-emission level-sweep result when readiness
   passes.
10. Project audit events for refused and completed attempts.
11. Sync outbox operations using entity type `simulated_test_execution`.
12. Strict idempotence for execution attempts through the existing operation
    journal.
13. Python local-agent client methods for execution launch and reads.
14. Qt action and form contract for one operator-centered workflow, not a
    scattered CRUD expansion.
15. Qt bootstrap/table support for execution history.
16. Real HTTP E2E coverage for refusal, correction, completion, replay,
    conflict, restart persistence, audit, and outbox.

## Model Notes

No deep metrology refactor was required in this slice. The metrology model
already separates calibration requirement, temporal validity, conformity,
serviceability, and planning reservation. The change made in `0.8.0` is to stop
treating readiness as an isolated instrument status at the execution boundary.

The execution workflow derives its own aggregate from metrology readiness:

- command context: project, method, attempt, mode, date, instruments;
- readiness report: copied from the metrology service;
- refusal: execution-specific and human-explainable;
- instrumentation snapshot: persisted per attempted execution;
- result: present only when the preflight passes.

Refused attempts are stored deliberately. A blocked preflight is laboratory
evidence and must remain traceable.

## Launch Commands

```text
cargo run -q -p emc-locus-agent -- storage init --storage-root data\agent --migrations-root storage\sqlite
cargo run -q -p emc-locus-agent -- serve --storage-root data\agent --migrations-root storage\sqlite --bind 127.0.0.1:8765
py apps\qt-console\main.py --agent-url http://127.0.0.1:8765 --migrations-root storage\sqlite
```

The Python CLI can also run the workflow:

```text
py -m emc_locus.gui_actions run-simulated-emc-test --agent-url http://127.0.0.1:8765 --attempt-id RUN-SIM-001 --project-code CEM-2026-001 --test-method-reference SIM-EMC-CONDUCTED --execution-mode accredited --asset-id SA-001 --operator operator.one --checked-on 2026-07-01 --reason "Operator launch" --operation-id op-run-sim-001
```

## Remaining Limits

- Only one simulated EMC test type is implemented.
- The result strategy is deterministic and synthetic.
- No real instrument control, DAQ, acquisition, FFT, uncertainty chain, report,
  file upload, object storage, planning reservation, central sync, PostgreSQL,
  Parquet, or HDF5 scope is included.
- The Qt view is still a temporary operator console, not the final Locus Test
  Station UI.

## Recommended Next Slice

The next slice should bind simulated execution to approved test definitions and
persist method parameters plus measurement-data evidence records for the
simulated result.
