# EMC Locus 0.7.0 - Metrology Readiness Vertical Slice

## Initial State

The release work started from the `0.6.0` project vertical slice architecture:

- Qt/Python uses a thin local-agent client;
- `emc-locus-agent` owns local HTTP routes and CLI commands;
- Rust services and repositories own business decisions and SQLite writes;
- writes produce revisioned audit and sync-outbox evidence;
- SQLite stays split by domain.

This release keeps that model and applies it to the first metrology vertical
slice. It does not replace the broader GUI, add a central server, or implement
real acquisition.

## Delivered

Version `0.7.0` consolidates the first agent-backed metrology readiness
vertical slice:

1. local storage initialization now includes the metrology database;
2. instruments can be registered, listed, and inspected through the Rust agent;
3. instrument identity includes asset ID, family, category, manufacturer,
   model, serial number, optional part number, calibration policy, configurable
   due-soon warning threshold, structured capabilities, notes, revisions, and
   serviceability state;
4. serviceability is separated from legacy availability and planning
   reservations through `usable`, `restricted`, `out_of_service`, and `retired`;
5. calibration events can be recorded with typed decisions and optional
   certificate document manifests;
6. certificate files are represented by metadata only, not stored as SQLite
   blobs;
7. calibration status is computed from the latest applicable event, decision,
   due date, checked date, calibration requirement, warning threshold, and
   serviceability state;
8. readiness can be assessed for one or more instruments in `accredited`,
   `non_accredited`, or `investigation` mode;
9. readiness responses include blocking issues, warnings, per-instrument
   reasons, revisions, serviceability, status, and calibration evidence;
10. migrated metrology writes produce audit rows, sync-outbox rows, and strict
    operation-id replay protection;
11. Python/Qt uses the agent path when `agent_url` is configured for migrated
    metrology reads and writes;
12. a real loopback HTTP E2E scenario verifies restart persistence, audit, and
    outbox behavior.

## Metrology Model

The model deliberately keeps three concerns separate:

- metrological fitness: calibration required, current computed status, and
  final decision such as `conforming` or `nonconforming`;
- operational serviceability: whether the asset can be used, restricted,
  quarantined, or retired;
- planning reservation: future scheduling ownership, preserved only as legacy
  `availability` data for now.

The computed statuses are:

```text
valid
due_soon
expired
missing
not_required
nonconforming
```

The default due-soon threshold is documented and can be overridden per
instrument through `calibration_due_warning_days`.

## Migration Strategy

Metrology migrations are non-destructive:

- existing `instruments` rows are preserved;
- legacy `availability=available` becomes serviceability `usable`;
- legacy `availability=out_of_service` remains `out_of_service`;
- legacy `availability=reserved` is treated as historical reservation data and
  does not make the instrument metrologically unavailable by default;
- legacy `calibration_records` rows are backfilled into `calibration_events`;
- original calibration rows and document references are preserved;
- migrations can be re-run without duplicating migrated calibration events.

Version 7 is the current metrology schema version.

## Qt Actions Migrated

When `agent_url` is configured, the temporary Qt/Python console no longer opens
`metrology.sqlite` directly for the migrated path:

- instrument list rows;
- instrument detail data;
- instrument registration;
- calibration-event recording with certificate manifest metadata;
- serviceability changes;
- computed status refresh;
- readiness display with ready/not-ready, blocking issues, warnings, and
  per-instrument status.

The standalone instrument-document form remains legacy until a dedicated agent
route exists.

## SQLite Structures

The slice uses or extends these local structures:

- `instruments`;
- `calibration_records`, preserved for historical compatibility;
- `calibration_events`;
- `instrument_documents`, still legacy for standalone attachments;
- `metrology_audit_events`;
- `sync_operations`;
- `schema_migrations`.

Future scientific data remains out of scope for SQL. Transducer correction
curves and large time-domain acquisitions should be stored outside SQLite in
appropriate scientific or tabular formats, with SQL keeping manifests, links,
versions, and checksums.

## Remaining Limits

- Standalone instrument document attachment still needs a dedicated agent route.
- Calibration-event audit rows are stored on the `calibration_event` entity;
  the current loopback audit route exposes instrument-entity audit rows.
- There is no object-storage upload or binary certificate transfer yet.
- There are no real drivers, acquisition, FFT, uncertainty chains, CEM test
  execution, reporting workflow, central synchronization, PostgreSQL, S3, HDF5,
  or Parquet implementation in this release.
- The Qt console remains a temporary validation surface, not the final advanced
  measurement GUI.

## Recommended Next Slice

The next vertical slice should be a first simulated EMC test execution that
selects a project and test definition, calls metrology readiness before
execution, refuses unsafe/accredited runs when readiness blocks, records a
simple simulated result, and produces traceable audit/outbox evidence.
