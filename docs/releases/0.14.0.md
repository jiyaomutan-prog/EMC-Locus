# Release 0.14.0 - General Equipment Inheritance And Physical Materials

Version `0.14.0` makes the Equipment Repository usable for shared laboratory
forms, local documentation, and first physical-material registration.

## Delivered

- An editable `Général` category is the universal inheritance root for all
  equipment families. Families retain their stable classification identity.
- Common manufacturer, model, internal-reference, description, and document
  rules are defined once at `Général` and resolved through effective templates.
- Repository Administration creates, edits, archives, assigns, and removes
  field rules. Required and optional states are explicit in administration,
  category creation, model creation, and preview.
- Equipment documentation is selected as a local file. The Rust agent stores
  the actual bytes by SHA-256 and returns a typed manifest referenced by the
  revisioned model definition.
- `Matériels réels` registers a serial-numbered laboratory asset through the
  existing metrology service from an approved equipment model, including
  calibration applicability, periodicity, warning window, and serviceability.
- The physical asset stores the approved equipment-model identity, revision,
  and canonical checksum without confusing equipment categories with the
  separate metrology taxonomy.
- Preview and diagnostics are one operator workflow; checksum and internal IDs
  remain available under `Informations techniques`.

## Model Decisions

- `Général` is an inheritance root, not a classification family and not a
  directly instantiable model category.
- Manufacturer and model name are protected structural identifiers. Other
  field definitions use archival deletion so historical snapshots remain
  readable.
- Serial number belongs to the physical material, not to the reusable model.
  The provisional model-level `Variante` field is therefore removed from the
  default form instead of being renamed.
- The Equipment Repository model reference is a typed three-part traceability
  contract on the metrology asset, not free text hidden in notes.
- SQLite stores the document manifest in canonical model JSON; binary payloads
  live in the local content-addressed object store below the storage root.

## Limits

- File upload uses a loopback JSON/base64 contract and a 20 MiB limit. Download,
  orphan collection, antivirus scanning, and remote object synchronization are
  not implemented yet.
- The physical-material workflow registers the metrology asset but does not yet
  add station connections, calibration-event entry in LAB CONSOLE, or a
  graphical measurement-chain builder.
- This release does not add real acquisition, FFT, reporting, authentication,
  RBAC, or final central synchronization.

## Validation

The release is validated with Rust formatting, Clippy and workspace tests;
Python compilation, unit tests and migration validation; LAB CONSOLE typecheck,
lint, unit tests, production build and Playwright E2E; release consistency and
Git whitespace checks. Browser inspection covers desktop and mobile Equipment
Repository administration and physical-material registration.
