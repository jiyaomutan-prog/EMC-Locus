# EMC Locus 0.6.1 - Metrology Serviceability Baseline

## Delivered

Version `0.6.1` starts the metrology vertical slice by separating three ideas
that were previously conflated:

1. metrological readiness;
2. operational serviceability;
3. planning reservation or legacy availability.

The metrology SQLite schema now keeps the historical `availability` column while
adding `serviceability_status`, `serviceability_reason`,
`serviceability_updated_at`, and `legacy_availability`.

## Migration Behavior

Existing instruments migrate non-destructively:

- `available` becomes serviceability `usable`;
- `reserved` becomes serviceability `usable` and remains traceable through
  `legacy_availability`;
- `out_of_service` becomes serviceability `out_of_service`.

This preserves older databases while preventing a planning reservation from
making an instrument metrologically unavailable by default.

## Operator Surface

- Qt view models and the static GUI now display service state separately from
  planning availability.
- The Python action layer exposes `set-instrument-serviceability`.
- The legacy `set-instrument-availability` action remains available and maps
  older out-of-service calls to the new serviceability state.

## Remaining Limits

- Metrology writes are still Python/SQLite local actions in this tranche; the
  Rust agent-backed metrology API starts in the next tranches.
- Calibration decision typing, readiness DTOs, audit/outbox, and HTTP E2E
  coverage remain targeted for the 0.7.0 metrology vertical slice.
