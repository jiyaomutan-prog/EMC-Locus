# Release 0.16.0 - Serial-Specific Signal Characterization

Version 0.16.0 closes the boundary left explicit in 0.15.1: a metrologist can
now record a correction measured for one physical inventory/serial number.

## Delivered

- A typed Rust core aggregate for immutable asset characterizations with
  canonical JSON and SHA-256 checksum.
- Two controlled result forms: time-sample conversion with gain, offset and
  optional overload/clipping bounds; frequency response with amplitude and
  optional phase over a measured frequency range.
- Metrology SQLite migration 9 with asset, validity, provider, method,
  decision, uncertainty, evidence manifest, recorder and revision evidence.
- Local-agent routes to record, list, retrieve and audit characterizations,
  plus content-addressed upload for metrology proof files.
- Atomic characterization, audit and pending outbox persistence with
  operation-id idempotence and replay mismatch detection.
- Python and TypeScript clients for the complete workflow.
- A LAB CONSOLE physical-asset dossier with status summary, characterization
  history, guided entry, applicability summary, evidence and progressively
  disclosed technical traceability.
- Real HTTP restart coverage and Playwright coverage from asset registration
  through proof upload, persistence, audit/outbox and reload.

## Domain Boundary

A reusable correction approved for an equipment model remains in
`equipment.sqlite`. A measured result for one serial number is an immutable
metrology event in `metrology.sqlite`. Recording a new measurement does not
replace the previous event.

The current test runtime does not select or apply these characterizations. A
future test-preparation workflow must pin the chosen characterization id and
checksum in the execution context.

## Operator Vocabulary

The normal interface uses `Signaux et corrections`, `Conversion temporelle`,
`Réponse fréquentielle`, and `Alimentation / conditionnement du capteur`.
Internal Rust names such as scaling profile or engineering curve remain
implementation contracts and are not presented as the operator's task.

## Limits

- No real acquisition, correction application, FFT, or measurement-chain
  runtime is added.
- No automatic choice between a generic model correction and an asset
  characterization is made.
- Calibration-event certificate bytes and standalone instrument documents are
  not migrated to the new upload route in this release.
- No authentication, RBAC, electronic signature, central synchronization, or
  compliance claim is added.
- Automated tests demonstrate software behavior, not laboratory fitness or EN
  ISO/IEC 17025 conformity.

## Validation

Release validation covers Rust formatting, clippy and workspace tests; Python
compilation, unit tests and SQLite migrations; LAB CONSOLE typecheck, lint,
unit tests, production build and six sequential Playwright workflows; exact
1440 x 900 and 1280 x 720 screenshots; launcher smoke tests; release
consistency and Git whitespace checks.
